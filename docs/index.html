Core Capabilities

Everything governance needs before mainnet

Tooling, configuration, and an inference engine designed to make AI service governance auditable and operationally honest.

🏛
Explainable Governance Inference
Classifies proposals as standard, high-impact, or safety-critical. Scores governance risk, determines approval paths, and produces a compact inference summary — no black-box decisions.
🔐
Signer Rotation & Audit Ledger
Full lifecycle for signer rotation: receipt, approval, finalization, activation, apply, verify — every step recorded in an append-only audit ledger with archive promotion and retained-baseline attestations.
🧠
Multi-Model Routing
Waterfall, consensus, hybrid, and specialized routing strategies. Reduces single-point-of-failure risk and enables flexible composition of deterministic and ML inference engines.
🌐
Permissioned Testnet
Reproducible local validator layout via render-localnet. Generates docker-compose, network-summary, and rendered genesis artifacts — no chain binary required to validate topology.

Identity Bootstrap

💰
Funding Deployment
Validates and generates deployment artifacts for treasury allocation, validator funding, and grant distribution across proportional, fixed, or dynamic funding pools.
Governance Engine

Inference that operators can challenge

Every proposal moves through a deterministic decision chain — every factor that drives a result is visible and auditable.

Proposal Classification

Risk scoring with explainable factors

The inference engine reads a proposal JSON document, classifies it, scores governance risk, and determines the required approval path (single-house, dual-house, safety-council escalation). Every output includes a compact inference summary with remediation items.

  • Standard · High-Impact · Safety-Critical classification
  • Risk score derived from deterministic policy rules
  • Required houses enumerated per classification
  • Machine-readable artifact output for external automation
governance-sim output 
{
  "proposal_class": "safety_critical",
  "risk_score": 0.87,
  "required_houses": [
    "token-house",
    "safety-council"
  ],
  "approval_path": "dual-house",
  "remediation": [
    "mandatory-postmortem",
    "emergency-pause-audit"
  ]
}
Drift & Trend Detection

Portfolio-level governance patterns

governance-drift compares a proposal against recorded history and flags repeated emergency use, adverse precedents, and treasury growth. governance-trends clusters the active queue so operators can spot systemic validator churn or recurring emergency governance across proposals.

  • Time-windowed baselines for seasonal pressure detection
  • Kind-level cadence vs. historical cluster comparison
  • Precedent clustering by proposal kind and requester
  • Structured mitigation bundles with severity and release blockers
governance-trends cluster 
"clusters": [
  {
    "kind": "emergency_pause",
    "count": 4,
    "seasonal_pressure": "above_normal",
    "severity": "high",
    "release_blocker": true,
    "owner": "safety-council-chair"
  }
]
Checkpoint Replay

Deterministic state from append-only logs

governance-replay reconstructs current checkpoint state from an event log or derived snapshot. Signed event logs are verified against signer policy — duplicate, contradictory, illegal, and out-of-order events are rejected at replay time.

  • Append-only event log with mandatory actor and rationale fields
  • HMAC signature verification per signer policy
  • Actor must be active in bootstrap identity registry
  • Transition-aware: only valid lifecycle moves accepted
checkpoint event 
{
  "checkpoint_id": "cp-2026-04-17-001",
  "previous_status": "in_progress",
  "new_status":      "completed",
  "actor_id":       "governance-chair-bot",
  "recorded_by":    "governance-ops-bot",
  "signature": {
    "format": "hmac-sha256",
    "signer_id": "governance-chair-bot"
  }
}

Roadmap

Milestone 1 — Narrow, executable scope

The first implementable chain milestone proves governed services can be registered, attested, and released without pretending to be a full public protocol surface.

Config & Policy Validation
Network topology, base genesis, and governance policy files are validated for internal consistency on every CI run.
Governance Tooling
Explainable inference engine, drift detection, trend clustering, remediation bundles, and checkpoint replay — all testable against example proposals without a running chain.
Signer Rotation Lifecycle
End-to-end signer rotation: receipt → approval → finalization → activation → apply → verify → ledger → archive, with retained-baseline attestations and promotion receipts.
Chain Modules (in progress)
registry, attestation, governance, and validator modules specified. Implementation sequence locked in config/modules/milestone-1.json.
Chain Binary & Runtime
No chain binary exists yet. This is the next major engineering phase after governance tooling is validated.
External Audit & Legal Review
Required before any public launch. Jurisdiction analysis, AML review, and sanctions screening are mandatory prereqs.
📦
Registry Module
Service records, release channels, and registry action log. Transactions: register_service, propose_release, retire_service.
✅
Attestation Module
Attestation records, reviewer bonds, and appeal windows. Transactions: submit_attestation, finalize_attestation, appeal_attestation.
🏛
Governance Module
Proposals, voting records, and execution hooks. Transactions: submit_proposal, cast_vote, execute_proposal.
🔲
Validator Module
Validator set management. Transactions: register_validator, update_validator, deactivate_validator.
